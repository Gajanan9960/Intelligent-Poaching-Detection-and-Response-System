Intelligent Poaching Detection and Response System - Demo Technical Q&A

=========================================
1. AI & MACHINE LEARNING (OBJECT DETECTION)
=========================================
Q: What model are you using for object detection, and why?
A: We are using YOLOv8 (specifically the nano version, yolov8n). It was chosen because YOLO (You Only Look Once) provides an excellent balance between high accuracy and real-time inference speed, which is critical for an early warning system. The 'nano' version ensures it can run efficiently even on edge devices or servers with limited compute.

Q: How does the model distinguish between a poacher, an animal, and a ranger?
A: The model has been trained on a custom dataset containing labeled images of animals, armed individuals (poachers), weapons, and vehicles. It detects features like the presence of weapons or specific clothing. (Note: Depending on your exact training set, it might classify any human with a weapon as a threat).

Q: How do you handle false positives, such as a tourist being flagged as a poacher?
A: The system relies on confidence thresholds. If a detection falls below a certain confidence score, it may just log it rather than trigger a critical alert. Additionally, human-in-the-loop verification (where an officer reviews the detection results) is used to filter out false positives before initiating a physical response.

=========================================
2. BACKEND ARCHITECTURE (FASTAPI & MONGODB)
=========================================
Q: Why did you choose FastAPI for the backend?
A: FastAPI was chosen because it is incredibly fast, natively supports asynchronous programming (async/await), and automatically generates interactive API documentation (Swagger UI). Its asynchronous nature is perfect for handling concurrent tasks like processing video streams, running ML inference, and serving API requests without blocking.

Q: Why use MongoDB instead of a relational database like PostgreSQL?
A: MongoDB, a NoSQL database, was chosen for its flexibility. The detection metadata generated by the AI model (bounding boxes, multiple detected classes per frame, varying confidence scores) is highly unstructured and hierarchical. Storing this as JSON-like BSON documents in MongoDB is much more efficient and schema-flexible than forcing it into rigid relational tables.

Q: How is the ML model integrated into the backend?
A: The model is loaded into memory when the FastAPI server starts. When a user or IoT camera uploads an image/video, the backend passes the file through the YOLOv8 model for inference, extracts the bounding boxes and classifications, saves the metadata to MongoDB, and returns the results to the frontend.

=========================================
3. FRONTEND & USER EXPERIENCE (REACT)
=========================================
Q: How are you managing state on the frontend?
A: We use React's built-in hooks (useState, useEffect) for local component state, and the Context API (AuthContext) for managing global state like user authentication and roles.

Q: Can you explain how Role-Based Access Control (RBAC) is implemented?
A: We implement RBAC on both the frontend and backend. 
- On the backend, we use a `RoleChecker` dependency in FastAPI that verifies the JWT token and checks if the user's role allows access to the requested endpoint.
- On the frontend, we protect routes in React Router based on the `user.role` (e.g., 'admin', 'officer', 'ranger'). We also conditionally render UI elements, like hiding the 'Analytics' dashboard from field rangers.

=========================================
4. SECURITY & DEPLOYMENT
=========================================
Q: How is the application secured?
A: The application uses JWT (JSON Web Tokens) for authentication. When a user logs in, they receive an encrypted token containing their identity and role. This token is sent in the Authorization header for all subsequent API requests. Passwords are also hashed in the database.

Q: What happens if the internet connection drops for a field ranger uploading an image?
A: (Assuming standard implementation) The frontend would catch the network timeout error and prompt the user to retry. For a production environment, we would implement offline caching or a Service Worker (PWA) to store the image locally and automatically sync it to the server once the connection is restored.
